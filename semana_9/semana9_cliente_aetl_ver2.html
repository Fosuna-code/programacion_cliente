⚠️ Perspectiva cliente — el Circuit Breaker vive en tu código, no en el servidor

El Circuit Breaker es un patrón que el cliente implementa para protegerse a sí mismo. El servidor no sabe que existe. No implementarás un proxy, un API gateway ni ningún componente de infraestructura — implementarás una clase Python o JS que envuelve tus llamadas HTTP y decide si ejecutarlas o no.

El problema que resuelve el Circuit Breaker

Cuando tu cliente hace retry automático ante fallos, está asumiendo que el servidor fallará ocasionalmente y que el reintento tendrá éxito. Esa suposición es correcta para fallos transitorios (sobrecarga momentánea, restart del servidor). Pero cuando el servidor está completamente caído, el retry se convierte en un problema: cada cliente sigue intentando, cada intento tarda el tiempo del timeout completo, y el servidor recibe una avalancha de peticiones justo cuando intenta recuperarse.

El Circuit Breaker resuelve esto con una idea de la ingeniería eléctrica: cuando hay un cortocircuito, el fusible "abre" el circuito para proteger el sistema. Cuando el fallo se resuelve, el circuito se "cierra" de nuevo. Traducido al software: cuando el servidor falla repetidamente, el cliente abre el circuito — falla inmediatamente sin contactar al servidor — y solo lo cierra cuando evidencia que el servidor se recuperó.

Los tres estados y sus transiciones

🟢 CERRADO
Operación normal. Peticiones fluyen al servidor. El cliente cuenta los fallos consecutivos.
🔴 ABIERTO
Circuito cortado. Las peticiones fallan inmediatamente con CircuitOpenError. No se contacta al servidor.
🟡 SEMIABIERTO
Prueba de recuperación. Una sola petición pasa. Éxito → Cerrado. Fallo → Abierto.

⚠️ Error común: contar 4xx como fallos del servidor

Si el Circuit Breaker cuenta los 401 Unauthorized como fallos del servidor, puede abrirse después de que el token expire — precisamente cuando el cliente más necesita comunicarse para renovarlo. Un 401 significa "tu token no es válido", no "el servidor está caído". La regla es clara: solo los errores de infraestructura (5xx, timeouts, errores de conexión) abren el circuito. Los errores de lógica del cliente (4xx) los maneja el interceptor HTTP y el TokenManager, no el Circuit Breaker.

💡 Degradación elegante — la UI no debe congelar

Cuando el Circuit Breaker está abierto, el cliente tiene tres opciones para la UI: (1) mostrar datos del caché local (si el ClienteSSEMultiplex tiene datos recientes), (2) mostrar un mensaje de "servicio no disponible" con el tiempo estimado de recuperación, o (3) deshabilitar las acciones de escritura (editar precio, ajustar stock) mientras permite ver los últimos datos conocidos. Las tres son mejores que congelar el panel esperando timeouts.

📐 Concepto arquitectónico: Jitter en el timeout de apertura

El timeout_apertura que elegiste (ej. 60s) es idéntico para todas las instancias del cliente. Si EcoMarket tiene 12 operadores con el panel abierto y el circuito se abre en todos al mismo tiempo, todos intentarán su petición de prueba al segundo 60 simultáneamente — una mini-avalancha sobre un servidor que acaba de recuperarse.

La solución es agregar jitter: un componente aleatorio al timeout para que cada instancia espere un tiempo ligeramente diferente.

# Python — timeout con jitter ±10%
import random
jitter = random.uniform(-0.1, 0.1) * timeout_apertura
timeout_efectivo = timeout_apertura + jitter # ej: 54s–66s

// JS — equivalente
const jitter = (Math.random() - 0.5) * 0.2 * timeoutApertura;
const timeoutEfectivo = timeoutApertura + jitter;

No se requiere implementarlo esta semana — pero un arquitecto de sistemas distribuidos debe saber que un timeout_apertura fijo sin jitter puede generar el "thundering herd problem" en sistemas multi-cliente. Si llegaste al Reto 8 (breaker por endpoint), pregúntale a la IA cómo combinar jitter con backoff exponencial para el estado de recuperación.

📖 Glosario — Semana 9

Circuit Breaker Pattern
Patrón de resiliencia del cliente que monitorea el éxito/fallo de las llamadas remotas y, ante fallo sostenido, "abre el circuito" rechazando peticiones temporalmente para proteger al servidor y al cliente de esperas indefinidas.
Estado Cerrado (Closed)
Estado normal del Circuit Breaker. Las peticiones fluyen al servidor. El breaker lleva conteo de fallos consecutivos pero no interviene hasta alcanzar el umbral.
Estado Abierto (Open)
El breaker rechaza peticiones inmediatamente con CircuitOpenError sin contactar al servidor. El servidor recibe cero presión mientras el cliente falla rápido.
Estado Semiabierto (Half-Open)
Transición controlada: una sola petición de prueba pasa al servidor. Éxito → circuito cerrado. Fallo → circuito abierto de nuevo con timeout completo reiniciado.
Fallo Rápido (Fail Fast)
Principio de diseño: es mejor fallar inmediatamente con un error claro que esperar el timeout completo. Mejora la experiencia del usuario y evita acumulación de peticiones pendientes en el cliente.
Degradación Elegante (Graceful Degradation)
Capacidad del cliente de ofrecer funcionalidad reducida (ej: datos del caché) en lugar de fallo total cuando el servidor no está disponible. El operador ve algo útil en lugar de una pantalla de error.
Retry con Backoff Exponencial
Estrategia donde el tiempo entre reintentos crece exponencialmente (1s, 2s, 4s, 8s...). Reduce la presión sobre servidores temporalmente saturados. Visto en Semana 2; esta semana se coordina con el Circuit Breaker.
Timeout de Apertura
Tiempo en segundos que el Circuit Breaker permanece ABIERTO antes de permitir una petición de prueba (estado SEMIABIERTO). Configurar demasiado corto impide la recuperación real del servidor.
Umbral de Fallos
Número de fallos consecutivos del servidor necesarios para que el Circuit Breaker transite a estado ABIERTO. Configurar demasiado bajo produce falsos positivos; demasiado alto permite que el servidor sufra demasiado antes de cortar.
Bulkhead Pattern
Patrón complementario al Circuit Breaker: limita el número de recursos concurrentes (conexiones, threads) dedicados a un servicio, evitando que un servicio lento consuma todos los recursos del cliente.
ClienteRobusto (AETL)
Denominación del componente cliente integrador de esta semana que combina CircuitBreaker + TokenManager + interceptor HTTP. Su principio rector: orquesta sin duplicar lógica de sus componentes.
Clasificación de Errores del Breaker (AETL)
Decisión de diseño explícita que determina qué errores abren el circuito (5xx, timeouts, errores de red) y cuáles no (4xx). Esta clasificación debe documentarse — no tomarse por omisión al codificar.
Deadlock de Auth-Breaker (AETL)
Anti-patrón: el Circuit Breaker abierto bloquea también las peticiones de refresh de token, dejando el sistema incapaz de autenticarse y de hacer peticiones simultáneamente. Se evita con un breaker separado para el endpoint de auth.
