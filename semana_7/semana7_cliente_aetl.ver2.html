🌐 El problema del pool de conexiones en el cliente

HTTP/1.1 limita el número de conexiones simultáneas que un cliente puede abrir hacia el mismo origen (mismo host + puerto + protocolo). Los navegadores modernos aplican un límite de 6 conexiones por origen; los clientes Python/Node.js no tienen ese límite nativo, pero cada conexión abierta consume un descriptor de archivo del sistema operativo y una ranura del pool TCP.

Una conexión SSE es persistente: permanece abierta indefinidamente mientras el servidor envía eventos. Cada EventSource en el navegador (o equivalente en Python/Node) ocupa una de esas 6 ranuras mientras exista. Con tres módulos escuchando simultáneamente, más los fetch() normales del panel, el pool se agota y las peticiones normales quedan esperando.

┌─────────────────────────────────────────────────────────────────────┐ │ CLIENTE (navegador / app) │ │ │ │ Pool de conexiones HTTP/1.1 hacia api.ecomarket.com │ │ ┌───────┬───────┬───────┬───────┬───────┬───────┐ │ │ │ SSE#1 │ SSE#2 │ SSE#3 │fetch()│fetch()│ ??? │ │ │ │precios│invent.│pedidos│ auth │ datos │[bloq.]│ │ │ └───────┴───────┴───────┴───────┴───────┴───────┘ │ │ ↑ ↑ │ │ Conexiones persistentes Los fetch() adicionales │ │ (no se liberan nunca) se encolan y esperan │ │ ← experiencia degradada │ └─────────────────────────────────────────────────────────────────────┘
📋 Qué ilustra este diagrama: Las tres conexiones SSE (una por módulo) ocupan permanentemente tres de las seis ranuras del pool del cliente. Cuando el código del cliente necesita hacer llamadas fetch() adicionales (renovar token, cargar datos históricos, enviar un pedido), esas peticiones se encolan porque el pool está lleno. Qué simplifica: omite el time-sharing de TCP y las variaciones por versión de navegador. Aspecto clave del protocolo: la naturaleza persistente de SSE convierte cada conexión en un recurso no-liberable mientras el cliente esté activo.

✅ La solución: multiplexación de eventos sobre una sola conexión

HTTP/1.1 y SSE permiten que el servidor envíe eventos de distintos tipos sobre la misma conexión usando el campo event:. En lugar de tres conexiones, el cliente abre una sola hacia /api/ecomarket/eventos?modulos=precios,inventario,pedidos y recibe todos los tipos de eventos mezclados en ese stream. El cliente es responsable de leer el campo event: y despachar cada mensaje al handler correcto. Eso es multiplexación del lado del cliente.

┌─────────────────────────────────────────────────────────────────────┐ │ CLIENTE (navegador / app) │ │ │ │ Pool de conexiones HTTP/1.1 hacia api.ecomarket.com │ │ ┌───────┬───────┬───────┬───────┬───────┬───────┐ │ │ │SSE │fetch()│fetch()│fetch()│fetch()│fetch()│ │ │ │multi- │ libre │ libre │ libre │ libre │ libre │ │ │ │plex │ │ │ │ │ │ │ │ └───────┴───────┴───────┴───────┴───────┴───────┘ │ │ │ │ │ │ GET /api/ecomarket/eventos?modulos=precios,inventario, │ │ │ pedidos │ │ ▼ │ │ ┌─────────────────────────────────────────────────────────┐ │ │ │ ClienteSSEMultiplex │ │ │ │ ┌──────────────────────────────────────────────────┐ │ │ │ │ │ EventRouter │ │ │ │ │ │ │ │ │ │ │ │ "precio-actualizado" ──► [handler_tabla_UI] │ │ │ │ │ │ "stock-critico" ──► [handler_alerta, │ │ │ │ │ │ handler_email] │ │ │ │ │ │ "pedido-nuevo" ──► [handler_pedidos_UI] │ │ │ │ │ │ "sistema-ping" ──► [handler_heartbeat] │ │ │ │ │ └──────────────────────────────────────────────────┘ │ │ │ └─────────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────────────┘
📋 Qué ilustra este diagrama: Una sola conexión SSE ocupa una ranura del pool; las otras cinco quedan libres para peticiones normales. El EventRouter vive completamente en el código del cliente — es una tabla de despacho en memoria. Qué simplifica: no muestra la reconexión ni el estado de la conexión. Aspecto clave del patrón: el routing por tipo de evento es responsabilidad del cliente; el servidor solo pone el campo event: en cada mensaje.

El protocolo SSE estándar (WHATWG) define los siguientes campos en cada bloque de evento. Esta semana el campo crítico es event::

ℹ️ ¿Qué pasa si el cliente no conoce el tipo de evento?

Si el servidor envía un event: sistema-alerta-critica que el cliente no tiene registrado en su EventRouter, el cliente debe ignorarlo silenciosamente — sin lanzar excepción, sin cortar la conexión. Esto permite al servidor agregar nuevos tipos de eventos sin forzar actualizaciones del cliente. El EventRouter pre-dado ya maneja este caso.

⚠️ Aclaración importante: cómo funciona Last-Event-ID — y cómo NO funciona

Un malentendido frecuente: el campo id: en un mensaje SSE no requiere que el cliente responda nada. SSE es unidireccional: el servidor envía, el cliente solo escucha. El cliente simplemente guarda en memoria el último ID visto (self.ultimo_id).

El Last-Event-ID solo se usa en la petición HTTP de reconexión — cuando la conexión se corta y el cliente abre una nueva. En ese momento, incluye el header Last-Event-ID: evt-005 para que el servidor sepa desde dónde reanudar. No existe ningún mensaje de confirmación del cliente hacia el servidor durante el stream normal.

── Stream normal (el cliente NO responde nada) ──────────────────
SERVIDOR → CLIENTE: id: evt-001 \n event: precio-actualizado \n data: {...} \n\n
SERVIDOR → CLIENTE: id: evt-002 \n event: stock-critico \n data: {...} \n\n
  ... (conexión TCP se corta aquí) ...

── Reconexión (el cliente envía Last-Event-ID en el GET) ────────
CLIENTE → SERVIDOR: GET /api/ecomarket/eventos?modulos=...
                    Last-Event-ID: evt-002   ← solo aquí aparece
SERVIDOR → CLIENTE: reanuda stream desde evt-003

⚙️ Cómo el cliente envía sus preferencias de suscripción

SSE usa HTTP GET para establecer la conexión. El cliente no puede enviar un cuerpo en un GET, por lo que las preferencias de suscripción se codifican como parámetros de URL. Esta es la responsabilidad clave del ClienteSSEMultiplex: construir la URL correcta antes de abrir la conexión.


# ── Pseudocódigo: Construcción de URL con suscripciones ─────────────────────
FUNCTION construir_url(base_url, modulos_activos):
    # modulos_activos es una lista: ["precios", "inventario", "pedidos"]

    # Codificar la lista como parámetro de URL
    # Resultado: ?modulos=precios,inventario,pedidos
    params = "modulos=" + UNIR(modulos_activos, separador=",")

    # Retornar la URL completa
    RETURN base_url + "?" + params

# ── Uso: ─────────────────────────────────────────────────────────────────────
url = construir_url(
    "https://api.ecomarket.com/eventos",
    ["precios", "inventario", "pedidos"]
)
# → "https://api.ecomarket.com/eventos?modulos=precios,inventario,pedidos"

# ── Si en tiempo de ejecución el cliente quiere AGREGAR un módulo: ────────────
# No puede simplemente agregar un handler — necesita cerrar la conexión actual,
# actualizar modulos_activos, y reconectar con la nueva URL.
# Esa decisión (¿cuándo reconectar?) es tuya como Arquitecto del Cliente.

🔄 Máquina de estados del cliente — Gestión de conexión

El ClienteSSEMultiplex debe rastrear en qué estado se encuentra su conexión para evitar abrir conexiones duplicadas o ignorar errores. Aquí el diagrama de estados que tu cliente debe implementar:

iniciar() │ ┌─────────────────▼──────────────────┐ │ DESCONECTADO │ │ (estado inicial y final) │ └─────────────────────────────────────┘ │ llamar a conectar() internamente │ ┌─────────────────▼──────────────────┐ │ CONECTANDO │ │ (solicitud HTTP en vuelo, │ │ esperando respuesta 200) │ └─────────────────────────────────────┘ │ │ HTTP 200 error/timeout/ stream inicia 4xx sin retry │ │ ┌────────────────────────▼──┐ ┌────────▼──────────────────────┐ │ CONECTADO │ │ DESCONECTADO │ │ (recibiendo eventos, │ │ (si error no recuperable: │ │ despachando al Router) │ │ 4xx, max reintentos) │ └────────────────────────┬──┘ └───────────────────────────────┘ │ │ detener() error 5xx / │ cierre TCP / │ timeout de red │ │ ┌────────▼──┐ ┌────────▼──────────────────────────────────────┐ │DESCONECTADO│ │ RECONECTANDO │ │(intencional│ │ (backoff exponencial, conteo de reintentos, │ │ por flag) │ │ Last-Event-ID enviado al reconectar) │ └────────────┘ └────────────────────────┬──────────────────────┘ │ conectar() internamente (regresa a CONECTANDO)
📋 Qué ilustra este diagrama: Los 4 estados posibles del cliente y cuándo ocurre cada transición. Por qué importa al código del cliente: si el cliente no verifica el estado antes de llamar a iniciar(), podría abrir una segunda conexión SSE mientras la primera sigue abierta — duplicando todos los eventos recibidos. El campo de estado (self.estado en Python) debe ser la única fuente de verdad. Simplificación: no muestra el manejo de concurrencia en entornos multihilo, solo el flujo lógico.

🔒 Invariantes que no deben romperse — COMPRENDE

  • INV-C1: El cliente no abre una segunda conexión SSE si ya hay una en estado CONECTADO o CONECTANDO. Verificar el estado antes de llamar a iniciar().

💡 Nota sobre la industria — Por qué no reimplementamos EventRouter

En producción, el EventRouter que aparece en este material es lo que hacen librerías como EventEmitter (Node.js core), RxJS (Angular/React), o PyDispatcher (Python). Lo importante no es saber reimplementar ese mecanismo de despacho genérico — es saber usarlo bien: qué tipo de evento registrar, qué hace tu handler cuando los datos son inesperados, y cómo manejar handlers que fallan sin romper el stream. Eso es lo que practicamos esta semana.

🖥️ Elige tu lenguaje/plataforma

El objective es el mismo en cualquier plataforma: una conexión HTTP GET persistente, leer líneas del stream, parsear el formato SSE y despachar al EventRouter. Cada opción enseña algo diferente:

  • Python + requests / aiohttp → lógica de stream más visible, ideal para ver el protocolo "puro". El parseo manual de líneas es explícito. Recomendado si vienes de Semana 6 con Python.
  • JavaScript/TypeScript + fetch + ReadableStream → event loop y async/await nativos, modelo de concurrencia web real. EventSource del browser ya lo hace, pero implementarlo con fetch enseña más.
  • Java/Kotlin + HttpClient → tipado fuerte, gestión explícita de null en los campos del evento, ideal para aprender manejo de datos inesperados.
  • CLI en cualquier lenguaje sin framework → ideal para ver el protocolo sin UI que distraiga. Verificas el comportamiento con cat validacion.log.

El objetivo no es copiar código — es entender qué decisión de diseño impone cada plataforma. ¿Por qué en Python necesitas iter_lines() y en JS necesitas un TextDecoder sobre el ReadableStream?

⚠️ Errores comunes esta semana

Error 1: Abrir conexión duplicada al reconectar
El cliente detecta un error de red, llama a iniciar() de nuevo sin verificar el estado, y termina con dos streams leyendo en paralelo. Cada evento llega duplicado. Solución: verificar estado == "DESCONECTADO" antes de todo iniciar(). La IA rara vez advierte sobre este race condition.
Error 2: Confundir línea vacía con evento sin datos
En el protocolo SSE, una línea vacía ("\n") es la señal de fin de bloque — significa "despacha el evento que venía construyendo". No es un evento vacío ni un error. Si el cliente dispara el router al recibir esa línea vacía sin haber parseado el tipo, se despacha con tipo=None y el router lo ignora... pero no debería llegar ahí. Solución: el router solo se llama cuando hay un campo event: en el bloque.
Error 3: No manejar el campo "event:" ausente
Si el servidor envía un bloque SSE sin campo event:, el protocolo define que el tipo es "message" por defecto. Si el cliente asume que siempre habrá un campo event: explícito, no registra ningún handler para "message" y silenciosamente descarta esos eventos. Solución: el tipo por defecto debe ser "message" si no aparece el campo event:.
Error 4: Resetear Last-Event-ID al reconectar
El cliente guarda el ID del último evento recibido para enviarlo como header Last-Event-ID al reconectar, permitiendo al servidor reanudar el stream desde donde se quedó. Si el cliente reinicia ese campo a None en cada reconexión, el servidor le vuelve a enviar todos los eventos desde el inicio y el cliente los procesa como nuevos. Solución: self.ultimo_id solo se resetea cuando el usuario llama explícitamente a detener().

🔒 Invariantes que no deben romperse — APLICA

  • INV-A1: El cliente nunca llama a router.despachar() con tipo=None. Si el bloque SSE no tiene campo event:, el tipo es "message" por defecto del protocolo.
  • INV-A2: Toda conexión HTTP debe configurar un timeout explícito. Sin timeout, una conexión colgada puede bloquear el cliente indefinidamente sin que el código lo detecte.
  • INV-A3: Un handler que lanza excepción no interrumpe la recepción de eventos ni afecta a otros handlers. El EventRouter captura la excepción y continúa. Este invariante es responsabilidad del EventRouter dado — no lo rompas modificando el código del router.
  • INV-A4: La variable self.ultimo_id no se resetea a None durante una reconexión automática (por fallo de red). Solo se resetea si el usuario explícitamente llama a detener() e iniciar() de nuevo desde cero.

🌐 HTTP/2 y el cliente SSE: ¿qué cambia en tu código?

HTTP/2 introduce multiplexación de streams a nivel de transporte: múltiples peticiones y respuestas viajan en paralelo sobre una única conexión TCP, identificadas por su stream ID. Esto significa que en teoría el límite de 6 conexiones por origen que resolviste esta semana con un solo SSE deja de ser el cuello de botella bajo HTTP/2.

Pero desde la perspectiva del código del cliente, ¿cuánto cambia realmente?

  • ✅ Lo que NO cambia: el parseo del stream text/event-stream, el manejo del campo event:, el EventRouter, los handlers. Todo eso es lógica de aplicación, independiente del transporte.
  • ⚠️ Lo que cambia para el cliente: en Python con httpx (que soporta HTTP/2), el cliente puede abrir múltiples streams concurrentes sobre la misma conexión TCP; no necesitas la multiplexación manual a nivel de URL que implementaste esta semana.
  • 🔍 Lo que el cliente debe decidir: ¿tres streams HTTP/2 separados (un EventSource por módulo) o un solo stream multiplexado a nivel de aplicación? Con HTTP/2, ambas opciones son igualmente eficientes en transporte. La elección depende de otros factores: ¿el servidor envía eventos correlacionados entre módulos? ¿Es más fácil para el cliente manejar streams independientes o el routing centralizado?

📖 Glosario — Semana 7

Multiplexación de eventos (cliente)
Técnica en la que un único stream SSE transporta mensajes de múltiples tipos de eventos, y el cliente los enruta al handler correcto usando el campo event:. Reduce las conexiones abiertas por el cliente a una sola.
Pool de conexiones HTTP/1.1
Conjunto limitado de conexiones TCP que el cliente (tipicamente el navegador, máximo 6 por origen) puede mantener simultáneamente hacia el mismo servidor. Cada SSE persistente ocupa una ranura de ese pool indefinidamente.
EventRouter (Dispatcher de eventos)
Componente del cliente que mantiene una tabla de tipo_evento → lista de handlers y ejecuta los handlers correspondientes cuando llega un evento. Un handler que falla no interrumpe los demás. En producción lo implementan librerías como EventEmitter (Node.js) o RxJS.
Suscripción filtrada
Mecanismo por el que el cliente indica al servidor, en la URL de la petición inicial, qué tipos de eventos quiere recibir (?modulos=precios,inventario). Reduce el volumen de datos que el cliente procesa sin necesidad de filtrar en el código cliente.
Máquina de estados del cliente
Modelo formal de los estados posibles del cliente SSE (DESCONECTADO, CONECTANDO, CONECTADO, RECONECTANDO) y las transiciones válidas. Evita condiciones de carrera como abrir conexiones duplicadas.
Last-Event-ID
Header HTTP que el cliente envía al reconectarse tras una interrupción. Contiene el ID del último evento recibido, permitiendo al servidor reanudar el stream sin reenviar eventos ya procesados. Es la "memoria de progreso" del cliente.
HTTP/2 multiplexación (impacto en cliente)
HTTP/2 multiplexa múltiples streams de petición/respuesta sobre una sola conexión TCP. Para el cliente SSE esto significa que el límite de 6 conexiones HTTP/1.1 desaparece a nivel de transporte, pero la lógica de EventRouter y handlers permanece igual.
Handler de evento
Función registrada en el EventRouter que el cliente ejecuta cuando llega un evento de un tipo específico. Cada tipo puede tener múltiples handlers. Un handler es independiente de los demás — su fallo no los afecta.
Andamiaje cognitivo (AETL)
Soporte temporal que el material o la IA proporciona para que el estudiante alcance comprensiones más allá de lo que lograría solo. En esta semana: el EventRouter pre-dado es andamiaje que libera carga cognitiva para enfocarse en la multiplexación.
Arquitecto del Cliente (AETL)
Rol que el estudiante mantiene durante la interacción con la IA: diseña antes de preguntar, toma las decisiones de configuración del cliente, y valida críticamente el código propuesto. La IA propone opciones y detecta casos frontera, pero no decide.
Caso frontera (AETL)
Condición de entrada o de red que el diseño inicial no contempla explícitamente: evento con datos no-JSON, tipo de evento desconocido, llamada a iniciar() con conexión activa. Validar casos frontera es el núcleo de la fase VALIDA.
